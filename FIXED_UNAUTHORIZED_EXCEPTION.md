# Fixed: UnauthorizedAccessException in Multiple Controllers

## Issue

When calling various API endpoints, the API was throwing:

```
System.UnauthorizedAccessException: Attempted to perform an unauthorized operation.
at BajajDocumentProcessing.API.Controllers.SubmissionsController.GetSubmission(Guid id, CancellationToken cancellationToken) in SubmissionsController.cs:line 88
```

## Root Cause

Multiple controller methods were trying to read the user ID from the "sub" claim:

```csharp
var userId = Guid.Parse(User.FindFirst("sub")?.Value ?? throw new System.UnauthorizedAccessException());
```

However, the JWT token uses `ClaimTypes.NameIdentifier` instead of "sub", causing the claim lookup to fail and throw an exception.

## Solution

Updated all affected methods to try multiple claim types for compatibility:

### Before:
```csharp
var userId = Guid.Parse(User.FindFirst("sub")?.Value ?? throw new System.UnauthorizedAccessException());
var userRole = User.FindFirst("role")?.Value;
```

### After:
```csharp
// Try both claim types for compatibility
var userIdClaim = User.FindFirst(ClaimTypes.NameIdentifier)?.Value ?? User.FindFirst("sub")?.Value;
if (string.IsNullOrEmpty(userIdClaim))
{
    _logger.LogWarning("User ID claim not found in token");
    return Unauthorized(new { error = "User ID not found in token" });
}

var userId = Guid.Parse(userIdClaim);
var userRole = User.FindFirst(ClaimTypes.Role)?.Value ?? User.FindFirst("role")?.Value;
```

## Related Issues

This is the same issue we fixed earlier in `DocumentsController`. The JWT token generated by the `AuthService` uses standard claim types:
- `ClaimTypes.NameIdentifier` for user ID
- `ClaimTypes.Role` for user role

All controllers now use these standard claim types with fallback to the string versions ("sub", "role") for compatibility.
